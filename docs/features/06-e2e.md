# Phase 6 — L4 e2e / behavior probe suite (where the grade lives)

> SPEC §2: *"The happy path is worth almost nothing; the grade lives in how failure is handled. Every
> probe maps to a judging dimension. They will run these tests."* This phase builds the probes that
> prove the system converges under failure. These are **L4** — they need a Docker daemon and the full
> compose stack (`make e2e`). `tests/e2e/` is currently empty.
>
> **Verified stack:** testcontainers **4.14.2** (`RabbitMqContainer.get_connection_params()` →
> pika params, not a URL; `PostgresContainer(driver="asyncpg")` for an async URL). For full-stack
> probes prefer driving the real `docker-compose` stack via `init.sh` + the worker `--scale`, killing
> containers with `docker kill`, over single-container testcontainers.

### R3.1 — Crash recovery probe   [rung R3.1] [scores: reliability ⭐]
depends_on: T1
files: `tests/e2e/test_crash_recovery.py`
context: `docker kill` a worker mid-job must lose **no** message; another worker (or the restarted one)
picks it up after the unacked message is redelivered, and the job converges to a correct final state.
This validates the ack-last ordering (W2) end-to-end.
steps:
  1. Submit a job with several blocks; once it's GENERATING, `docker kill` the (or a) worker.
  2. Assert the job still reaches COMPLETED with the full asset (redelivery recovered the in-flight task).
MUST: assert **no message loss** and a correct terminal state (not a partial drama marked COMPLETED).
verify: [L4] `make e2e -k crash_recovery`.
accept: killed-mid-job → redelivered → COMPLETED, asset intact.

MUST: assert the global concurrent-TTS peak respects the limit (soft/best-effort per X5 — log breaches),
  and that a crashed holder's slot is reclaimed (no deadlock-to-0).
verify: [L4] `make e2e -k semaphore`.
accept: ≤3 concurrent across 4 workers; killed holder's slot returns.

### R4.2 — Cache + fan-in probe   [rung R4.2] [scores: state ⭐]
depends_on: T1
files: `tests/e2e/test_cache_fanin.py`
context: Two identical blocks in one job → the vendor is hit **once** (the 2nd is a cache hit that burns
no slot), yet both remain distinct task rows that each decrement — so the job still produces exactly one
`StitchReady`. This is the "cache-vs-counter must not be conflated" probe (SPEC §4) plus the fan-in
exactly-once.
steps:
  1. Submit a manuscript with two identical blocks (+ some distinct ones).
  2. Assert exactly one vendor synthesis for the duplicate hash (e.g. count `tts/<hash>.wav` writes or a
     sim call counter), AND exactly one `StitchReady` / one COMPLETED with all blocks represented.
MUST: assert one vendor call for identical blocks (Constraint B) AND that both tasks still decremented
  (two task rows, one cache entry) — exactly one StitchReady.
verify: [L4] `make e2e -k cache or -k fan_in`.
accept: dedup at the vendor, not at the counter; single StitchReady.

### E-EDGE — Edge-case battery   [rung R4.x] [scores: edge ⭐]
depends_on: T1
files: `tests/e2e/test_edge_cases.py`
context: The edge cases SPEC §3 explicitly calls out beyond the 5 named failures: **0-block** and
**1-block** manuscripts (fan-in must still terminate), **cache-hit-meets-fan-in** (a job where every
block is a cache hit still completes), **parse-crash-after-writing-some-task-rows** (re-run via ON
CONFLICT completes without duplicates), and **dependency-down-mid-job** (kill Redis/MinIO briefly →
the job recovers or fails cleanly, never corrupts).
steps:
  1. 0-block manuscript → terminates (STITCHING/COMPLETED), no hang.
  2. 1-block manuscript → completes with one chunk.
  3. All-cache-hit job → completes with exactly one StitchReady, zero new vendor calls.
  4. Kill the worker after parse wrote some rows but before fan-out finished → redelivery completes the
     job once (no duplicate tasks).
  5. Bounce a dependency (e.g. `docker restart redis`) mid-job → job converges to a correct state.
MUST: every edge case converges to a correct terminal state — none hangs, none corrupts.
verify: [L4] `make e2e -k edge`.
accept: all five edge scenarios terminate correctly.
